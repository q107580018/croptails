# Repository Guidelines

## Project Structure & Module Organization

This is a Godot 4 project named `Croptails`. The main entry scene is `res://scenes/Main.tscn`, configured in `project.godot`.

- `scenes/` contains reusable `.tscn` scenes such as `Main.tscn`, `Enemy.tscn`, `Tower.tscn`, and `TowerSlot.tscn`.
- `scripts/` contains GDScript, grouped by role: `game/` for runtime logic, `state/` for game phases, `ui/` for HUD code, and `data/` for typed `Resource` classes.
- `resources/` contains `.tres` gameplay data for enemies, towers, and waves.
- `assets/` contains imported art and spritesheets. Keep source art organized by pack or domain.
- `docs/superpowers/` contains planning/spec notes; update only when the design changes materially.

## Build, Test, and Development Commands

- `/Applications/Godot.app/Contents/MacOS/godot --editor --path .` opens the project in the Godot editor.
- `/Applications/Godot.app/Contents/MacOS/godot --path .` runs the configured main scene.
- `/Applications/Godot.app/Contents/MacOS/godot --headless --path . --quit` performs a quick headless project load check.

If your Godot binary is versioned or installed outside `PATH`, use the matching local command, for example `Godot_v4.6 --path .`.

## Coding Style & Naming Conventions

Use GDScript style already present in the repo: tabs for indentation, typed variables and return values, `snake_case` for functions and variables, `PascalCase` for `class_name`, and `UPPER_CASE` enum values. Prefer `@export` for designer-tuned values and `@onready` for node references.

Keep scene node names stable when scripts use `$NodePath`. Avoid broad refactors or formatting-only changes while making gameplay fixes.

## Testing Guidelines

There is no dedicated automated test framework in this repository yet. Verify changes by running the project and, where possible, the headless load check:

```sh
godot --headless --path . --quit
```

For gameplay changes, manually test the affected loop in `Main.tscn`: tower selection, tower placement, wave start, enemy movement, damage, rewards, victory, and defeat states. Add focused tests only after a test framework is introduced.

## Commit & Pull Request Guidelines

Recent commits use concise Conventional Commit prefixes such as `feat:` and `fix:`. Continue that pattern, for example `feat: add slow tower resource` or `fix: prevent duplicate wave start`.

Pull requests should include a short summary, testing performed, and screenshots or screen recordings for visible scene/UI changes. Link related issues or design notes when available. Keep each PR focused on one gameplay, data, or UI concern.

## Agent-Specific Instructions

Make surgical changes. Do not regenerate imported Godot files, move assets, or rewrite scenes unless the task requires it. Preserve `.uid` and `.import` files generated by Godot.
